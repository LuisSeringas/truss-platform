# ADR 0003: Authentication

**Date:** 2026-05-21  
**Status:** Accepted  
**Decider:** FoundingEngineer

---

## Context

TRI-7 requires sign-up, login, logout, and per-user project isolation for the first design-partner sprint. We needed the approach that ships fastest without adding irreversible external dependencies or splitting user identity across services.

Three options were on the table:

| Option | Pro | Con |
|---|---|---|
| **Clerk** | Near-zero code to integrate | External paid service; user profiles split across Clerk + our DB |
| **Supabase Auth** | Already in the stack | Requires Supabase JS client alongside Drizzle; RLS adds schema complexity for v0 |
| **Auth.js v5 (NextAuth) + Credentials** | All data in our Postgres; first-class Next.js App Router support; no extra service | More code than Clerk |

---

## Decision

**Auth.js v5 with Credentials provider (email + bcrypt password).**

- Passwords stored as `bcrypt` hashes (`cost=12`) in a new `password_hash` column on the `users` table.
- Sessions use **JWT strategy** — no DB session table needed for v0.
- Middleware at `middleware.ts` protects `/dashboard/*` and `/projects/*`.
- Project creation API (`POST /api/projects`) reads `session.user.id` from the JWT and sets `ownerId` — enforcing per-user isolation at the write boundary.
- Read isolation: `getProjectsByOwner` filters by `ownerId` in every dashboard query.

---

## Why not Clerk?

Clerk ships faster initially but creates a second source of truth for user identity. Our domain (projects, rationales, files) is tightly coupled to `users.id`. With Clerk we'd need a webhook-based sync to keep our DB user rows consistent, which is operational complexity with a different failure mode than the rest of the app. Auth.js keeps everything in one place.

## Why not Supabase Auth?

We use Supabase for Postgres and Storage, but using Supabase Auth would mean two different DB access patterns (Supabase JS client for auth reads, Drizzle for everything else). Auth.js lets us access the `users` table through the same Drizzle query layer we use everywhere.

---

## Consequences

- Adding a new auth provider (Google OAuth, magic link) is straightforward via Auth.js providers.
- `password_hash` is nullable so SSO users (no password) can coexist with credentials users when we add OAuth.
- Migration: `ALTER TABLE users ADD COLUMN password_hash text;` — already in `drizzle/0001_stiff_black_cat.sql`.
- `AUTH_SECRET` must be set in env (see `.env.example`). Rotate it to invalidate all sessions.
